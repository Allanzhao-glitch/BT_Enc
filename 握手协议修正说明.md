# 握手协议修正说明

## 问题分析

之前的实现中,握手数据被错误地包裹在 BluFi 协议头中,导致车端无法识别和处理握手请求。

### 错误的实现 (之前)

```
握手数据流:
  KeyMatchHeader (50字节)
  ↓ 错误地包裹 BluFi
  4D 04 00 32 [KeyMatchHeader 50字节]
  ↓ 发送
  车端收不到或无法识别
```

### 参考 ble_client.py 的正确实现

查看 `BT_Enc/radar/ble_client.py` 第50-63行:

```python
def send_key_change_data(data):
    # 握手阶段直接发送,不包裹 BluFi
    hex_encode = aes256.aes256_encrypt(data, key, iv)
    data = bytes.fromhex(hex_encode)
    client_pub_len = len(data)
    
    if communication_stage == 1:
        is_change = 20
    
    # 直接构造 KeyMatchHeader
    header = struct.pack('BB', client_pub_len, is_change)
    struct_data = header + data
    
    # 直接发送,不经过 BluFi 封装!
    blue.send_data(struct_data)
```

**关键发现**: 
- `send_key_change_data()` 直接发送 `KeyMatchHeader`,**不使用 BluFi 封装**
- 只有通讯阶段 (`communication_stage == 1`) 的数据才使用 BluFi 协议 (第235-281行 `send_data()` 方法)

## 修正方案

### 协议分层

```
握手阶段 (communication_stage = 0):
  KeyMatchHeader (原始数据) → 直接发送

通讯阶段 (communication_stage = 1):
  加密数据 → BluFi 封装 → 发送
```

### 修改内容

#### 1. 第1次握手 - 发送客户端公钥

**修改前**:
```python
request = await self.crypto_client.handshake_step1_send_public_key()

# 错误: 包裹 BluFi
blufi_seq = self.get_next_blufi_seq()
blufi_packet = self.create_blufi_packet(request, blufi_seq, packet_type=0x04)

await self.client.write_gatt_char(UUIDS.WRITE, blufi_packet, response=True)
```

**修改后**:
```python
request = await self.crypto_client.handshake_step1_send_public_key()

# 正确: 直接发送 KeyMatchHeader,不包裹 BluFi
logger.info(f"KeyMatchHeader: {len(request)} 字节")
logger.info(f"完整包(hex): {request.hex()}")

await self.client.write_gatt_char(UUIDS.WRITE, request, response=True)
```

#### 2. 第3次握手 - 发送确认

**修改前**:
```python

# 错误: 包裹 BluFi
blufi_seq = self.get_next_blufi_seq()
blufi_confirm = self.create_blufi_packet(confirm, blufi_seq, packet_type=0x04)

await self.client.write_gatt_char(UUIDS.WRITE, blufi_confirm, response=True)
```

**修改后**:
```python

# 正确: 直接发送确认数据,不包裹 BluFi

logger.info(f"完整包(hex): {confirm.hex()}")

```

#### 3. 接收处理 - 支持两种模式

**修改后**:
```python
def _notification_handler(self, sender, data: bytearray):
    logger.info(f"接收通知数据: {len(data)} 字节, hex: {data.hex()}")
    
    # 尝试解析 BluFi 包
    is_blufi, seq, payload = self.parse_blufi_packet(data)
    
    if is_blufi and payload:
        # 通讯阶段: BluFi 包装的加密数据
        logger.info(f"接收BluFi载荷: {len(payload)} 字节 (seq={seq})")
        self.receive_buffer.extend(payload)
    else:
        # 握手阶段: 原始 KeyMatchHeader
        logger.info(f"接收原始数据: {len(data)} 字节")
        self.receive_buffer.extend(data)
    
    self.receive_event.set()
```

#### 4. 加密数据发送 - 仍然使用 BluFi

```python
async def send_encrypted_data(self, data: bytes) -> bool:
    # 加密数据
    encrypted = await self.crypto_client.encrypt_send_data(data)
    
    # 通讯阶段使用 BluFi 协议封装
    blufi_seq = self.get_next_blufi_seq()
    blufi_packet = self.create_blufi_packet(encrypted, blufi_seq, packet_type=0x04)
    
    await self.client.write_gatt_char(UUIDS.WRITE, blufi_packet)
```

## 数据流对比

### 握手阶段

**修改前 (错误)**:
```
PC → 车端:
  4d 04 00 32 [KeyMatchHeader 50字节]
  │  │  │  │
  │  │  │  └─ 长度(50)
  │  │  └─ 序列号(0)
  │  └─ 类型(0x04)
  └─ BluFi头(0x4D)

车端: 收不到或无法识别! ❌
```

**修改后 (正确)**:
```
PC → 车端:
  [KeyMatchHeader 50字节]
  30 00 [加密公钥 48字节]
  │  │
  │  └─ is_change (0x00 = 第1次握手)
  └─ client_pub_len (48)

车端: 正确识别为握手请求! ✅
```

### 通讯阶段

```
PC → 车端:
  4d 04 02 [len] [EncryptedDataHeader + 加密数据]
  │  │  │
  │  │  └─ 序列号(2)
  │  └─ 类型(0x04)
  └─ BluFi头(0x4D)

车端: 解析 BluFi 包,提取加密数据! ✅
```

## 总结

### ✅ 正确的协议使用

| 阶段 | 发送格式 | 接收格式 |
|------|---------|---------|
| 握手第1次 | KeyMatchHeader (原始) | - |
| 握手第2次 | - | KeyMatchHeader (原始) |
| 握手第3次 | 确认数据 (原始) | - |
| 通讯阶段 | BluFi[EncryptedDataHeader] | BluFi[EncryptedDataHeader] |

### ❌ 之前的错误

握手阶段也使用了 BluFi 封装,导致车端无法识别。

### 🎯 修正结果

- 握手数据直接发送,不包裹 BluFi
- 通讯数据使用 BluFi 封装
- 接收处理自动识别两种格式
